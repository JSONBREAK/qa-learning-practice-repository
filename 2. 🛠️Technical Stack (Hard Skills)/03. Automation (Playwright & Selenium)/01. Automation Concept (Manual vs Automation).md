**Objective:** เข้าใจนิยามของ Test Automation, ข้อดี-ข้อเสีย และการเลือก Test Case ที่เหมาะสมในการทำ Automation (ROI Thinking)

---

## 1. Manual vs. Automation (ต่างกันยังไง?)

|**คุณสมบัติ**|**Manual Testing**|**Automation Testing**|
|---|---|---|
|**ความเร็ว**|ช้า (ขึ้นอยู่กับคน)|เร็วมาก (รันได้ 24/7)|
|**ความแม่นยำ**|มีโอกาสเกิด Human Error|แม่นยำสูงตามที่ Code สั่ง|
|**ค่าใช้จ่าย**|ต่ำในตอนแรก (ค่าแรงคน)|สูงในตอนแรก (ค่า Tool/คนเขียน)|
|**การทดสอบซ้ำ**|น่าเบื่อและใช้เวลา|ทำได้ง่ายและรวดเร็ว (Regression)|
|**ความยืดหยุ่น**|ปรับตัวตาม UI ที่เปลี่ยนได้ทันที|Code พังง่ายถ้า UI เปลี่ยน (Maintenance)|

---

## 2. จะทำ Auto เมื่อไหร่? (The Automation Pyramid)

เราไม่ควรทำ Automation 100% เพราะมันใช้เวลาดูแลรักษาสูง ให้ยึดหลัก **Test Pyramid**:

- **Unit Tests:** (ฐานกว้างสุด) เน้นทดสอบ Logic เล็กๆ โดย Dev
    
- **API / Service Tests:** ทดสอบการเชื่อมต่อข้อมูล (ที่เราเรียนไปใน Postman)
    
- **UI / E2E Tests:** (ยอดพ้นน้ำ) ทดสอบหน้าจอเหมือนที่ User ใช้จริง **(กิ่งนี้เราจะโฟกัสตรงนี้!)**
    

---

## 3. ควรเลือก Test Case ไหนมาทำ Automation? (Selection Criteria)

ห้ามทำ Auto ทุกเคส! ให้เลือกจากหลักการ **"High Value, Low Maintenance"**:

1. **Regression Tests:** เคสที่ต้องรันซ้ำทุกครั้งที่มีการ Update ระบบ
    
2. **Smoke Tests:** เคสสำคัญที่เช็คว่าระบบหลักยังไม่พัง (เช่น Login, Checkout)
    
3. **Complex Calculation:** เคสที่ต้องคำนวณตัวเลขเยอะๆ ซึ่งคนอาจพลาดได้
    
4. **Data-Driven:** เคสที่ต้องใช้ข้อมูลชุดเดิมรันซ้ำหลายๆ แบบ (เช่น ลอง Login ด้วย 100 User)
    

---

## 4. สิ่งที่ Automation ทำไม่ได้ (หรือทำได้ไม่ดี)

- **UX/UI Aesthetic:** การดูว่าสีสวยไหม ตัวหนังสือเบี้ยวไหม (คนดูเก่งกว่า)
    
- **Exploratory Testing:** การเดินเล่นในระบบเพื่อหาบั๊กใหม่ๆ (ต้องใช้สัญชาตญาณคน)
    
- **One-time Test:** เคสที่รันแค่ครั้งเดียวแล้วทิ้ง (ไม่คุ้มเวลาเขียน Code)
