## 🎯 Bug Triage คืออะไร?

  

**Bug Triage** คือการประชุมเพื่อ

**คัดกรอง จัดลำดับความสำคัญ และตัดสินใจจัดการบั๊กอย่างเป็นระบบ**

  

เป้าหมายคือให้ทีม:

- ใช้เวลาและทรัพยากรได้คุ้มค่า

- แก้บั๊กที่กระทบผู้ใช้และธุรกิจมากที่สุดก่อน

  

> 🎯 เป้าหมายหลัก:

> **ตัดสินใจให้เร็ว ชัด และตรงจุด**

> ว่าบั๊กไหนควรทำก่อน ใครรับผิดชอบ และต้องแก้เมื่อไหร่

  

---

  

## 🧩 ทำไม Bug Triage ถึงสำคัญ?

  

- จำนวนบั๊ก > เวลาและจำนวนคนที่มี

- ป้องกันการ **Fix ผิดลำดับ** (แก้เรื่องเล็กก่อนเรื่องใหญ่)

- ลดความสับสนระหว่าง QA / Dev / PM

- ช่วยให้การ Release มีความเสถียร

  

> ถ้าไม่มี Bug Triage → ทีมจะ “ยุ่ง” แต่ไม่ “คืบหน้า”

  

---

  

## 👥 ผู้เข้าร่วม Bug Triage (Typical Roles)

  

- **QA / Tester**

- อธิบายอาการบั๊ก

- วิธี Reproduce

- ผลกระทบต่อผู้ใช้ / ระบบ

  

- **Developer**

- ประเมิน Effort (ความยากง่าย)

- แนวทางการแก้ไข

- ความเสี่ยงเชิงเทคนิค

  

- **Product Owner / PM**

- ตัดสิน Priority

- มองผลกระทบเชิงธุรกิจ / Timeline

  

- **Tech Lead (ถ้ามี)**

- ช่วยตัดสินเคสที่ซับซ้อน

- มองผลกระทบเชิงโครงสร้างระบบ

  

---

  

## 🔄 กระบวนการ Bug Triage (Flow การประชุม)

  

1. **Review Bug List**

- ดูบั๊กใหม่

- บั๊กค้าง

- บั๊กที่ใกล้ Release

  

2. **Clarify & Reproduce**

- เข้าใจอาการให้ตรงกัน

- เช็คเงื่อนไขที่เกิดบั๊ก

  

3. **Assess Severity**

- ผลกระทบต่อระบบ / ผู้ใช้

- (QA เป็นคนประเมินหลัก)

  

4. **Assess Priority**

- ควรแก้เมื่อไหร่

- (PO / PM ตัดสิน)

  

5. **Decide Action**

- Fix now

- Fix later

- Won’t fix

- Duplicate

  

6. **Assign Owner & Timeline**

- ใครรับผิดชอบ

- แก้เมื่อไหร่

## ⚖️ Severity vs Priority (หัวใจของ Bug Triage)

```
Severity (Impact)  
▲  
│  
Effort ─── Priority
```


### 🔴 Severity (ผลกระทบทางเทคนิค)

- **Critical** – ระบบล่ม / ใช้งานไม่ได้ทั้งหมด / ข้อมูลเสียหาย
- **High** – ฟังก์ชันหลักพัง แต่มีทางเลี่ยง
- **Medium** – ฟังก์ชันรองมีปัญหา
- **Low** – Cosmetic / UI เล็กน้อย

📌 *QA เป็นผู้ประเมิน Severity*

---

### 🔵 Priority (ความเร่งด่วนเชิงธุรกิจ)

- **P0** – ต้องแก้ทันที ก่อน Release
- **P1** – แก้ใน Sprint นี้
- **P2** – เลื่อนได้
- **P3** – Backlog

📌 *PO / PM เป็นผู้ตัดสิน Priority*

> ⚠️ หมายเหตุสำคัญ  
> **Severity สูง ≠ Priority สูงเสมอ**  
> (ขึ้นกับบริบทธุรกิจและ Timeline)

---

## 🧠 คำถามสำคัญที่ใช้ตัดสินใจใน Bug Triage

- ผู้ใช้เจอปัญหานี้บ่อยแค่ไหน?
- กระทบรายได้ / ลูกค้า / ภาพลักษณ์หรือไม่?
- มี Workaround ไหม?
- Effort ในการแก้สูงแค่ไหน?
- ถ้าไม่แก้ตอนนี้ จะเสี่ยงอะไร?

## 🛠️ RCA ในบริบทของ Bug Triage

> RCA **ไม่ใช่ทุกบั๊กต้องทำ**  
> แต่จะถูกหยิบมาใช้ใน Bug Triage เมื่อ:

- บั๊กมี **Severity สูง**
- เป็นบั๊ก **ซ้ำ**
- บั๊ก **หลุด Production**
- กระทบ Process ระยะยาว

---

### 🔍 เครื่องมือ RCA ที่ใช้บ่อย

#### 1️⃣ The 5 Whys

ใช้ถาม “ทำไม” ไล่ลงไปจนถึงต้นเหตุ

ตัวอย่าง:
- ปัญหา: ระบบค้างหน้า OTP
- Root Cause: ไม่มีขั้นตอน Database Review

> การแก้ที่ถูกต้อง = แก้ **Process** ไม่ใช่แค่ **Symptom**

---

#### 2️⃣ Fishbone Diagram

ใช้เมื่อปัญหาซับซ้อน มีหลายปัจจัย

หมวดหลัก (ตัวอย่าง):
- People
- Method
- Machine
- Material

---

## 📤 ผลลัพธ์ที่ต้องได้จาก Bug Triage (Output)

หลังจบ Bug Triage  
**ทุกบั๊กควรมีอย่างน้อย:**

- Status ชัดเจน
- Severity & Priority ที่ตกลงแล้ว
- Owner
- Timeline คร่าว ๆ
- Decision (Fix / Defer / Reject / Duplicate)

---

## ✅ Best Practices สำหรับ Bug Triage

- บั๊กต้องเขียนชัด Reproduce ได้
- แยก Severity กับ Priority ให้ชัด
- ตัดสินใจจากข้อมูล ไม่ใช่ความรู้สึก
- ทุกบั๊กต้องมี Owner และสถานะ

---

> 📝 **สรุปท้ายไฟล์:**  
> Bug Triage ไม่ใช่แค่การ “จัดบั๊ก”  
> แต่คือ **การตัดสินใจเชิงคุณภาพ**  
> เพื่อให้ทีมส่งซอฟต์แวร์ที่ดีที่สุดภายใต้ข้อจำกัดที่มี
