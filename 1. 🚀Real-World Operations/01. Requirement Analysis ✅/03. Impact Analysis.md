# 🔄 Impact Analysis — วิเคราะห์ผลกระทบจากการเปลี่ยนแปลง Requirement

**สถานะ:** 🟡 Draft / ใช้ร่วมกับ Requirement Analysis  
**ใช้เมื่อ:** Requirement เปลี่ยน / เพิ่ม Feature / แก้ Bug ใหญ่ / ก่อนรับงานเข้า Sprint

---

## 🎯 Impact Analysis คืออะไร?

**Impact Analysis** = การวิเคราะห์ว่า  
> “ถ้าเปลี่ยนสิ่งนี้ → อะไรจะโดนกระทบบ้าง”

เพื่อให้ทีม:
- ไม่หลุด Requirement เดิม
- ไม่พัง Feature อื่นโดยไม่รู้ตัว
- ประเมินความเสี่ยง / เวลา / ความยากได้แม่นขึ้น

> Impact Analysis ≠ การออกแบบ  
> แต่คือการ **ประเมินผลกระทบก่อนลงมือ**

---

## 🧠 QA ทำ Impact Analysis ไปเพื่ออะไร?

QA ใช้ Impact Analysis เพื่อ:
- หา **Regression Area**
- ชี้จุดเสี่ยงที่ต้องเทสเพิ่ม
- เตือนทีมว่ามีอะไร “ซ่อนอยู่” ใต้ Requirement ใหม่

---

## 📌 Trigger ที่ต้องทำ Impact Analysis

ทำทันทีเมื่อ:
- 🔁 Requirement เปลี่ยน (Change Request)
- ➕ เพิ่ม Feature ใหม่
- 🐞 แก้ Bug ที่แตะ Core Logic
- 🚀 เตรียม Release ใหญ่
- ⚠️ Bug หลุด Production

---

## 🧩 มุมมองในการทำ Impact Analysis (QA Focus)

### 1️⃣ Functional Impact
- Feature ไหนทำงานเปลี่ยน?
- Flow ไหนเปลี่ยน?
- Logic เดิมยังใช้ได้ไหม?

ตัวอย่าง:
- เพิ่ม Limit การโอน → Validation เดิมยังถูกไหม?
- เปลี่ยน Rule → Test Case เดิม Fail ไหม?

---

### 2️⃣ Non-Functional Impact
- Performance ช้าลงไหม?
- Security เปิดช่องโหว่ไหม?
- Reliability / Recovery ยังโอเคไหม?

ตัวอย่าง:
- เพิ่ม TPS → ต้องเทส Load เพิ่ม
- เพิ่ม Role ใหม่ → สิทธิ์เข้าถึงเปลี่ยน

---

### 3️⃣ Data Impact
- Schema DB เปลี่ยนไหม?
- Data เดิมยังใช้งานได้?
- ต้อง Migration ไหม?

---

### 4️⃣ Integration Impact
- กระทบ API ไหน?
- Third-party ยังใช้ Contract เดิมได้ไหม?
- Version compatibility?

---

### 5️⃣ Test Impact (มุม QA ล้วน)
- Test Case ไหนต้อง:
  - Update
  - Add
  - Retire
- ต้อง Regression Scope แค่ไหน?

---

## 🛠️ Impact Analysis Template (QA ใช้จริง)

Change Description:

- เพิ่ม Limit การโอนเงินจาก 50,000 → 100,000 บาท
    

Impacted Areas:

- Feature: Transfer Money
    
- Validation Logic
    
- Daily Limit Calculation
    

Functional Impact:

- ต้องแก้ Validation amount
    
- Error message ต้องเปลี่ยน
    

Non-Functional Impact:

- Performance: ไม่มี
    
- Security: ต้องเช็ค Fraud Rule
    

Data Impact:

- ไม่มีการเปลี่ยน Schema
    

Integration Impact:

- Backend Transfer API
    
- Fraud Detection Service
    

Test Impact:

- Update Test Case: TC-TR-001 ถึง TC-TR-005
    
- Add New Test Case: Boundary (99,999 / 100,000 / 100,001)
    
- Regression: Transfer + Transaction History
    

Risk:

- High (แตะ Core Money Flow)

## 📊 Impact Analysis แบบตาราง (อ่านเร็ว)

| Area | Impact | Risk |
|-----|-------|------|
| Feature | Transfer Flow | High |
| Validation | Amount Limit | High |
| API | Transfer API | Medium |
| DB | No change | Low |
| Test Case | Update + Add | High |

## 🔍 คำถามที่ QA ควรถามตอนทำ Impact Analysis

- อะไรเปลี่ยน “โดยตรง”?
    
- อะไรอาจพัง “ทางอ้อม”?
    
- Flow ไหนเคยพังมาก่อน?
    
- เคสไหนควร Regression เสมอ?
    
- มี Edge case ใหม่เกิดไหม?


---

## ⚠️ Pitfall ที่พบบ่อย

| ❌ พลาด | ทำไมอันตราย |
|-------|-------------|
| ดูแค่ Feature ใหม่ | Bug หลุด Feature เก่า |
| ไม่คิด Regression | Bug โผล่ตอน Release |
| ไม่ถาม Non-functional | ระบบช้า / ล่ม |
| ไม่ Update Test Case | เทสไม่ตรงของจริง |
