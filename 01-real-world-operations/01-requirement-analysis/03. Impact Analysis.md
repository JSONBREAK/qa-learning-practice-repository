# 🔄 Impact Analysis — วิเคราะห์ผลกระทบจากการเปลี่ยนแปลง Requirement

**สถานะ:** ✅Done
**ใช้เมื่อ:** Requirement เปลี่ยน / เพิ่ม Feature / แก้ Bug ใหญ่ / ก่อนรับงานเข้า Sprint

---

## 🎯 Impact Analysis คืออะไร?

**Impact Analysis** = การวิเคราะห์ว่า  
> “ถ้าเปลี่ยนสิ่งนี้ → อะไรจะโดนกระทบบ้าง”

เพื่อให้ทีม:
- ไม่หลุด Requirement เดิม
- ไม่พัง Feature อื่นโดยไม่รู้ตัว
- ประเมินความเสี่ยง / เวลา / ความยากได้แม่นขึ้น

> Impact Analysis ≠ การออกแบบ  
> แต่คือการ **ประเมินผลกระทบก่อนลงมือ**

---

## 🔍 Visualizing the Ripple Effect (การมองความเชื่อมโยง)

เมื่อมีการเปลี่ยนแปลงเกิดขึ้น ให้มองผลกระทบเป็น 3 วงกว้างๆ:

1. **Direct Impact:** ฟีเจอร์ที่แก้โดยตรง (เช่น แก้ปุ่มโอนเงิน)
2. **Indirect Impact:** ฟีเจอร์ที่อยู่ติดกันหรือใช้ Data ร่วมกัน (เช่น ประวัติการโอนเงิน, ยอดเงินคงเหลือ)
3. **Cross-Platform Impact:** ผลกระทบต่อระบบอื่นหรือ Third-party (เช่น API ที่ส่งข้อมูลไปสรรพากร)

## 📌 Trigger ที่ต้องทำ Impact Analysis

ทำทันทีเมื่อ:
- 🔁 Requirement เปลี่ยน (Change Request)
- ➕ เพิ่ม Feature ใหม่
- 🐞 แก้ Bug ที่แตะ Core Logic
- 🚀 เตรียม Release ใหญ่
- ⚠️ Bug หลุด Production

## 🧩 มุมมองในการทำ Impact Analysis (QA Focus)

### 1️⃣ Functional Impact (กระทบการทำงาน)

ดูความเชื่อมโยงของ Business Flow

**ตัวอย่าง:**  แก้ไข Logic "การคำนวณภาษี" → กระทบไปถึง "หน้าตะกร้าสินค้า", "ใบเสร็จ PDF", และ "รายงานสรุปยอดขายรายเดือน"

---

### 2️⃣ Non-Functional Impact
- Performance ช้าลงไหม?
- Security เปิดช่องโหว่ไหม?
- Reliability / Recovery ยังโอเคไหม?

ตัวอย่าง:
- เพิ่ม TPS → ต้องเทส Load เพิ่ม
- เพิ่ม Role ใหม่ → สิทธิ์เข้าถึงเปลี่ยน

---

### 3️⃣ Data Impact
- Schema DB เปลี่ยนไหม?
- Data เดิมยังใช้งานได้?
- ต้อง Migration ไหม?

>  **Migration** คือ
> การ  “ย้าย / ปรับโครงสร้าง / เปลี่ยนข้อมูลเดิม  
> ให้เข้ากับโครงสร้างหรือกติกาใหม่ของระบบ”

---

### 4️⃣ Integration Impact
- กระทบ API ไหน?
- Third-party ยังใช้ Contract เดิมได้ไหม?
- Version compatibility?

---

### 5️⃣ Test Impact (มุม QA ล้วน)
- Test Case ไหนต้อง:
  - Update
  - Add
  - Retire
- ต้อง Regression Scope แค่ไหน?

- Long-term risk:

## 📊 Summary: Impact Level Decision Matrix

| **Impact Level**       | **คำนิยาม**                               | **Action สำหรับ QA**                               |
| ---------------------- | ----------------------------------------- | -------------------------------------------------- |
| **🔴 High (Critical)** | กระทบ Core Flow, การเงิน, หรือ Security   | **Full Regression** (เทสทั้งระบบ) + New Test Cases |
| **🟡 Medium**          | กระทบฟีเจอร์ย่อย หรือ Logic ในหน้าเดียว   | **Partial Regression** (เทสเฉพาะ Module ที่เกี่ยว) |
| **🟢 Low**             | แก้คำผิด, เปลี่ยนสีปุ่ม, หรือ UI เล็กน้อย | **Sanity Test** (เช็คแค่จุดที่แก้และส่วนใกล้เคียง) |

## 🔍 คำถามที่ QA ควรถามตอนทำ Impact Analysis

#### 1️⃣ใครโดนกระทบ (WHO)
- ผู้ใช้กลุ่มไหนเจอปัญหานี้?
- User เก่า / ใหม่ / role พิเศษ?
- มี % คร่าว ๆ ไหม?
👉 ถ้าตอบไม่ได้ = impact ยังไม่ชัด

#### 2️⃣ ผู้ใช้ “ทำอะไรไม่ได้” (WHAT)
- เป็น core flow หรือ edge case?
- ถ้าเป็น user เราจะรู้สึกยังไง?
👉 คำว่า “พัง” ต้องแปลงเป็นพฤติกรรมที่เกิดขึ้น

#### 3️⃣ ขอบเขตปัญหาอยู่ตรงไหน (SCOPE)
- เกิดเฉพาะ environment ไหน?
- เกิดตลอด หรือบางเงื่อนไข?
- เกิดหลัง event อะไร (deploy / config / data)?
- ลามไป feature อื่นไหม?

#### 4️⃣ ความรุนแรงจริง ๆ คืออะไร (SEVERITY)
- ผู้ใช้ยังใช้งานระบบต่อได้ไหม?
- ถ้าใช้งานไม่ได้ กระทบ flow หลักไหม?
- มี data loss หรือ risk ด้าน security ไหม?
- กระทบรายได้ / SLA / compliance ไหม?
❗ Severity ≠ ความยากในการแก้

#### 5️⃣ ถ้าไม่รีบแก้ จะเกิดอะไรขึ้น (RISK)
- จะกระทบความเชื่อมั่นไหม?
- มีโอกาสพังซ้ำใน flow อื่นไหม?
- ปัญหาจะลุกลามไหม?
#### 6️⃣ ควรเร่งระดับไหน (PRIORITY)
- ควรหยุด release ไหม?
- ต้อง hotfix ไหม?
- รอ release รอบถัดไปได้ไหม?
- QA ควร block ไหม?

---

## ⚠️ Pitfall ที่พบบ่อย

| ❌ พลาด | ทำไมอันตราย |
|-------|-------------|
| ดูแค่ Feature ใหม่ | Bug หลุด Feature เก่า |
| ไม่คิด Regression | Bug โผล่ตอน Release |
| ไม่ถาม Non-functional | ระบบช้า / ล่ม |
| ไม่ Update Test Case | เทสไม่ตรงของจริง |

## 🔗 การเชื่อมโยงและความสัมพันธ์

- **Input:** นำ Logic จาก **Requirement Analysis** และเงื่อนไขจาก **AC** มาเป็นตัวตั้งต้นเพื่อดูว่าของเดิมกับของใหม่ขัดกันไหม
    
- **Output (Design):** ส่งต่อขอบเขตที่วิเคราะห์ได้ไปให้ **Test Case Design** เพื่อเขียนเคสใหม่และอัปเดตเคสเก่าให้ตรงตาม Impact
    
- **Output (Strategy):** ใช้ระดับความรุนแรง (Impact Level) ไปกำหนดขอบเขตใน **Regression Testing** ว่าต้องเทสซ้ำมากน้อยแค่ไหน
    
- **Maintenance:** ข้อมูลจากขั้นตอนนี้จะถูกนำไปอัปเดตใน **RTM** เพื่อให้รู้ว่า Requirement นี้ผูกกับ Test Case ชุดไหนบ้างในปัจจุบัน

## 💡 Key Takeaway

- Impact Analysis คือ **เกราะกัน Bug หลุด**
- QA ที่ดี = เห็นผลกระทบก่อนคนอื่น
- ไม่ต้องละเอียดทุกครั้ง แต่ **ต้องไม่มองข้าม**

> เปลี่ยนนิดเดียว อาจพังทั้งระบบ  
> Impact Analysis คือสิ่งที่กันเรื่องนี้ได้ดีที่สุด
